## [high][performance] Bundled U2Netp cut-out and all mask post-processing run on the UI isolate

**Where:** `lib/features/segmentation/engines/bundled/bundled_segmentation_engine.dart:56`

The bundled engine's segment() does img.decodeImage of the full-resolution photo (pure-Dart JPEG decode of a 2048x2048 source is hundreds of ms to seconds on a midrange phone), copyResize, _rgbBytes, MaskTensor.packTensor, and MaskTensor.unpackMask (bilinear upscale of the 320x320 map to up to 4.2M pixels) with no Isolate.run anywhere. Downstream, editor_screen.dart:1173 then runs MaskProcessing.process (threshold + BFS connected-components over the full-res mask + separable box blur) on the UI isolate too, and the tier-1 tap-to-remove at editor_screen.dart:1291 calls MaskProcessing.removeObjectAt, which labels every 4-connected component of the full source-res mask (Int32List(n) + BFS queue over ~4M px) per tap, followed by feather + subtract, plus the per-pixel overlap loop in _samRemoveAt (editor_screen.dart:1348). Every one of these freezes the raster thread's frame pump for the duration — the 'Working…' spinner itself cannot animate. MobileSamEngine already demonstrates the correct pattern in this codebase (Isolate.run for _prepareInput at mobile_sam_engine.dart:179 and _lowResToMask at line 141).

**Recommendation:** Wrap the whole BundledSegmentationEngine pre/post pipeline (decode -> resize -> pack, and unpackMask) in Isolate.run around the infer() call, and move MaskProcessing.process / removeObjectAt / subtract+feather / the overlap loop into Isolate.run as well — AlphaMask is just ints + Uint8List, so it crosses isolates cheaply, matching what mobile_sam_engine.dart already does.

**Verifier:** Confirmed by direct code inspection. BundledSegmentationEngine.segment() (bundled_segmentation_engine.dart:54-97) performs pure-Dart decodeImage, copyResize, _rgbBytes, MaskTensor.packTensor and MaskTensor.unpackMask (bilinear upscale of the 320x320 map to full source resolution, mask_tensor.dart:95-116) with no Isolate.run; a grep of all of lib/ shows Isolate.run appears only in mobile_sam_engine.dart:141 and :179, exactly the contrasting pattern the finding cites. The registry (segmentation_registry.dart:47) invokes engine.segment() directly from _removeBackground in editor_screen.dart:1157, so all of it runs on the UI isolate. Downstream, editor_screen.dart:1173 runs MaskProcessing.process (BFS keep-largest-component over the full-res mask, mask_processing.dart:101-146, plus separable box-blur feather, lines 64-96) on the UI isolate; editor_screen.dart:1291 calls MaskProcessing.removeObjectAt which labels every 4-connected component with Int32List(n) + BFS queue (mask_processing.dart:201-221) then feather+subtract per tap; and editor_screen.dart:1348 is the per-pixel overlap loop in _samRemoveAt. The 'Working…' overlay (editor_screen.dart:832, 243) is animated by the same isolate. The codebase itself acknowledges the hazard in mobile_sam_engine.dart:176-177: "Pure-Dart decode/resize/pack off the UI isolate (a 2048² decode + HWC pack is real jank territory on the main thread)" — the bundled engine does precisely that work on the main thread.

**Corrections:** Minor inaccuracies: (1) MaskProcessing.process at editor_screen.dart:1173 is called with const MaskProcessingOptions(), whose threshold defaults to null, so the threshold step is skipped — the actual UI-isolate work there is keepLargestComponent (BFS) + feather, not "threshold + BFS + blur". (2) The UI isolate is not blocked for the whole end-to-end duration in one stretch: segmenter.infer() is awaited across a native boundary, so the freeze occurs in two chunks (decode/resize/pack before infer, unpackMask + MaskProcessing after) — still each long enough to freeze the spinner, but the finding slightly overstates it as one continuous block. (3) The timing figures (hundreds of ms to seconds, 2048x2048 source) are plausible estimates, not measured in the code; unpackMask upscales to whatever the decoded image size is. All structural claims and every cited line number are accurate.

## [high][ai-usage] No runtime capability probe: SAM and bundled-engine isAvailable() are constant-true asset checks; nothing gates the ~+104 MB encoder on low-RAM devices

**Where:** `lib/features/segmentation/engines/object/mobile_sam_engine.dart:75`

MobileSamEngine.isAvailable (lines 75-83) only does rootBundle.load of the two .onnx assets, and BundledSegmentationEngine.isAvailable (bundled_segmentation_engine.dart:44-51) only loads u2netp.onnx. All three assets ship in every build (pubspec.yaml:85-90), so both checks are constant-true on device — they verify packaging, never runtime viability (ORT session creation, ABI/native-lib load, available RAM). There is no RAM or capability probe anywhere in lib/ (grep for MemoryInfo/isLowRamDevice/device_info returns nothing). Consequences: (a) on a 2-3 GB RAM device the TinyViT encoder pass (~+104 MB RSS, plus full-resolution img.decodeImage of the source photo in _prepareInput) can push the process into an Android low-memory-killer kill — the app simply vanishes mid-edit, and no Dart catch block can intercept that; (b) if ORT session creation fails (missing/incompatible native lib), the failure is only discovered inside _samRemoveAt's generic catch (editor_screen.dart:1368) and is never remembered — every subsequent tap re-pays the full decode+resize+encoder attempt before failing again with 'Couldn't remove that — try again', an infinite retry invitation for a feature that can never work.

**Recommendation:** Add a memoized aiCapabilityProvider probed once per app start: (1) extend the existing sticker_maker/platform MethodChannel (lib/core/platform/platform_services.dart) with a getMemoryInfo call returning ActivityManager.getMemoryInfo().totalMem/availMem and isLowRamDevice(); (2) verify ORT works by creating the SAM decoder session once (it stays resident during use anyway) and caching failure permanently. Gate the tier-2 escalation in _samRemoveAt and the 'Remove object' tab on this provider — on isLowRamDevice or totalMem below ~3 GB, keep only the free connected-component tier and steer users to Erase. Also consider downscaling _prepareInput's initial decode for very large photos to cap the decode-side allocation.

**Verifier:** Verified line-for-line: MobileSamEngine.isAvailable (mobile_sam_engine.dart:75-83) and BundledSegmentationEngine.isAvailable (bundled_segmentation_engine.dart:44-51) are pure rootBundle.load asset checks, and pubspec.yaml:85-90 ships all three .onnx assets in every build, so both are constant-true on device — they never probe ORT session creation, native-lib load, or RAM. A project-wide case-insensitive grep for MemoryInfo/isLowRamDevice/device_info/totalMem/availMem/ActivityManager/lowRam returns zero hits, so no capability or memory probe exists anywhere. The retry claim also holds: in _ensureEmbedding (mobile_sam_engine.dart:160-204) the full readAsBytes + Isolate.run(_prepareInput) full-resolution decode/resize/pack executes BEFORE _graphFactory(encoderAsset), the embedding disk cache is written only on success, no failure state is ever cached, and the only user-facing handler is the generic catch (_) at editor_screen.dart:1368 ("Couldn't remove that — try again") — so a permanently broken ORT runtime costs the full decode+encoder attempt on every tap forever. The precompute warm-up (editor_screen.dart:876-880, mobile_sam_engine.dart:86-92) swallows the same failure silently, reinforcing that nothing remembers it.

**Corrections:** The "~+104 MB RSS" encoder figure is an unsourced estimate not derivable from the repo: the mobile_sam_encoder.onnx asset is only ~14.4 MB on disk, and the engine already caps resident footprint by disposing the encoder session immediately after its single pass (mobile_sam_engine.dart:181-203, comment says session+arena is "tens of MB") — a mitigation the finding does not acknowledge. Minor addition strengthening the finding: the failure is not only discovered in _samRemoveAt — the eager precompute triggered when the "Remove object" tab is armed (editor_screen.dart:876-880) attempts the same encoder pass and silently swallows the error via catch (_) in precompute. All file/line references in the finding are otherwise accurate.
